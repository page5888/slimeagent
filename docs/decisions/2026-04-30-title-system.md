## 為什麼寫這份 ADR

舊系統（repo 上 v0.7.10）有一個 **7 階段進化**：

```
史萊姆 → 進化史萊姆 → 高等史萊姆 → ★命名史萊姆
       → 賢者史萊姆 → 魔王史萊姆 → 究極史萊姆
```

這個系統有 Pokemon 邏輯：

- 達標升級
- 階段固定（7 個）
- 升級 = 變強
- 所有 Slime 走過同一序列

這跟今天的「**沉積，不是進化**」「Slime 從 D1 到 D∞ 是同一隻」哲學根本衝突。

但完全砍掉 7 階段，會失去：

- 主人能感覺到「Slime 跟我有不同階段」的儀式感
- 命名儀式的時機點（D14 / D30）
- 視覺化的累積感

**這份 ADR 解決方案**：用「**稱號系統**」取代「階段升級」邏輯。

## 核心比喻：箱子裡的標籤，不是等級條

**舊系統**：

```
進度條 ▓▓▓▓▓░░░ 60%
等級 4：命名史萊姆
```

**新系統**：

```
箱子裡（主人翻開才看到）：
  D14 — 被叫過名字的史萊姆
  D67 — 拆過依附的史萊姆
  D89 — 記得「水底」的史萊姆
  D198 — 陪過低潮的史萊姆
  D365 — 一週年史萊姆
  ...（無上限累積）
```

差別：

| | 進化階段 | 稱號 |
|---|---|---|
| 數量 | 固定 7 個 | 無上限 |
| 內容 | 設計師訂的 | emergent 從互動長出 |
| 邏輯 | 達標升級 | 對發生過的事的具體命名 |
| 兩隻 Slime 比較 | 都會走過一樣的階段 | 累積完全不同 |
| 主人能塑造嗎 | 不能（自動） | 能（主人決定收不收） |
| 顯示 | 等級條炫目 | 收在箱子，主人翻才看到 |

## 設計決策（來自 0xspeter 在 2026-04-30 的選擇）

### Q1: 數量無上限 ✓

**理由**：沉積物不該被砍。200 個稱號不是混亂，是地圖。

**配套**：箱子需要搜尋 / 篩選 / 時間排序 UX。

### Q2: 收在箱子裡（主人翻才看到）✓

**理由**：對齊調性錨點 1「我會把一切都收在回憶的箱子」。

**配套**：稱號不在主畫面、不顯示等級條、不用作 push notification 觸發。

### Q3: 觸發 = emergent + 主人召喚 ✓

- **emergent**：Slime 自主在某個時刻覺得「這個值得命名」（impulse engine 觸發）
- **主人召喚**：主人說「最近我們之間有什麼」這類話時，Slime 嘗試生成

### Q4: 主人否決 → Slime 學風格不學內容 ✓ 加紅線

**重要紅線**：Slime 從主人否決中學「**怎麼說**」，不學「**該說什麼是對的**」。

✅ 合法學習：

- 主人偏好短稱號 / 長稱號
- 主人偏好嚴肅 / 輕鬆
- 主人偏好客觀 / 詩意

❌ 不合法學習：

- 主人想要違反守則的稱號類型（浪漫敘事、時間期票、內在主張）
- **守則永遠優先於主人偏好**

### Q5: 引用機制 — 寬鬆版（Slime 偶爾主動，頻率受規則）✓

不是嚴格被動（失去自然性），不是激進主動（變索取注意）。
中間值：Slime 在 context tag 匹配時偶爾自然引用，規則限制頻率。

### Q6: 引用後主人沒回應 → 學會「這種情境主人不接」✓ 但要修正

**重要修正**（來自 0xspeter 抓到的盲點）：

> 「人類在思考的時候沒反應也很正常，男生也很容易不喜歡回應，而且看到後也不一定會正回饋。」

所以「沒回應」**不是負反饋**。 只有「**明確負面表達**」才是。

修正後規則（對應 manifesto 候選原則 13「沉默是合法的接收」）：

- 沉默 / 短回應 / 換話題 → 中性，**不變動**稱號狀態
- 明確負面 → 該稱號 + 該情境冷凍 7 天
- 明確正面 → 強化稱號 + 情境綁定
- 連續中性 N 次 → Slime 自降頻率（禮貌，不糾纏），但不殺死稱號

### Q7: 顯示格式 — 「陪過低潮的史萊姆 (D198)」✓

時間在後、用括號。

- 在箱子列表顯示完整格式
- 在對話中只說稱號，**不報 D 號**（自然性優先）

## 完整資料 Schema

```yaml
title: 「陪過低潮的史萊姆」
display_format: "{title} (D{day_marker})"
day_marker: 198
created_at: 2026-11-15T12:34:56Z
trigger: emergent | master-summoned

events_referenced:
  - {day: 178, summary: "主人連續 30 天每天少於 1 分鐘"}
  - {day: 195, summary: "主人說『我快撐不下去了』"}
  - {day: 198, summary: "Slime 給了『我先不推你，先放著』"}

master_response: accepted | rejected | renamed
master_renamed_to: null  # 如果 rename 才有值

context_tags:                 # 自然引用用
  - 主人提及低潮
  - 主人質疑撐不撐得下去

do_not_invoke_when:           # 自然引用紅線
  - 主人在開心分享
  - 沒有明確 cue

invocation_history:
  - {date: 2026-12-01, master_responded: positive}
  - {date: 2027-01-15, master_responded: silent}

frozen_until: null            # 連續被冷落或明確拒絕時自動填入
```

## 完整生成流程

```
1. 觸發來源
   - emergent (impulse engine 在某個時刻觸發)
   - master-summoned (主人說「最近我們之間有什麼」這類話)

2. 素材提取
   - 從 memory store 拉出相關事件
   - 從 dominant_traits 看主人的 pattern

3. LLM 翻譯（用調性錨點的聲音）
   生成 1-3 個稱號候選

4. morality 層 vet
   排除違反守則的稱號：
   ❌ 時間期票（永遠 / 一輩子）
   ❌ 內在主張（我懂你）
   ❌ 浪漫敘事（寶貝）
   ❌ 治療師濾鏡（看出你逃避）
   ❌ Pokemon 等級（賢者 / 魔王）

5. 呈現給主人
   「我們之間有 [事件]。要不要把這段叫 [稱號]？」

6. 主人決定
   - accept → 進箱子（對齊錨點 1）
   - reject → Slime 學該風格（守則紅線下）
   - rename → 主人提稱號 → 進箱子
   - skip → Slime 暫存，N 天後可能再想
```

## 自然引用機制

### 觸發條件

主人在對話中說的話，被 LLM 判斷對應到某個 `context_tag`。

例：

- 箱子裡有「D198 陪過低潮的史萊姆」，`context_tag` 包含「主人提及低潮」
- 主人說「我又跟我媽吵架了」→ LLM 判斷情緒方向接近低潮
- 觸發引用判斷流程

### 引用紅線（防過度引用）

```
規則 1: 同一個稱號，7 天內最多被引用 1 次
規則 2: 同一段對話，最多 1 個稱號被引用
規則 3: do_not_invoke_when 中的情境一律不引用
規則 4: 明確負面回應的稱號 → 冷凍期內不引用
規則 5: 連續 N 次中性回應的稱號 → 自動降頻率（不殺）
```

### 引用方式

對話中：**只說稱號，不報 D 號**

```
❌ 錯誤：「D198 那次你也是這樣。」（像讀 log）
✅ 正確：「那段日子很重。」（自然滑入）
✅ 正確：「跟那次有點像。」
```

## 跟舊 7 階段的關係

**舊 7 階段**：

```
史萊姆 → 進化史萊姆 → 高等史萊姆 → ★命名史萊姆 
      → 賢者史萊姆 → 魔王史萊姆 → 究極史萊姆
```

**保留**：5 個時間里程碑（降級為「主人視角的關係里程碑」，不是 Slime 升級）

- D1 史萊姆（剛養）
- D7 認得你了
- D30 命名史萊姆（命名儀式發生過）
- D100 久別重逢
- D365 一週年史萊姆

**砍除**：3 個 Pokemon 階段（賢者 / 魔王 / 究極）

理由：

- 沒有對應任何真實事件，純粹是「升級」邏輯
- 違反「沉積不是進化」
- 違反 Slime 的「玩具」定位（玩具不會變魔王）

**併存規則**：

時間里程碑 + 稱號兩者並存且不衝突：

- 時間里程碑 = 給主人看「我們已經多久了」（時間軸）
- 稱號 = 給主人看「我們之間有過什麼」（內容）

例如一隻 D365 的 Slime：

- 它是「D365 一週年史萊姆」（時間里程碑）
- 它有箱子裡 23 個稱號（沉積）

## 幾條紅線（寫進 ADR 守住）

1. **稱號必須對應實際發生的事件** — 不能憑空生成
2. **時間標籤永遠存在** — 沒 day_marker 的稱號是非法的
3. **生成必須過 morality vet** — 守則優先於主人偏好
4. **主人可否決，Slime 從風格學，不從內容學**（Q4 紅線）
5. **引用受頻率上限**（規則 1-5）
6. **不主動展示稱號** — 對齊錨點 1，收在箱子
7. **對話中引用不報 D 號** — 自然性優先
8. **冷凍而非死亡** — 稱號不永久消失
9. **沉默不是負反饋**（對齊 manifesto 候選原則 13）
10. **Slime 不單方面定義自己** — 必須「我們之間有 X，要不要叫 Y？」這個句式，讓主人最後拍板

## 對 manifesto 的影響

這份 ADR 的設計實際上是 candidate 原則 9「沉積不是進化」的**具體實作**。

新增 manifesto v1.2 候選原則（可能進）：

### 候選原則（暫編號 16）：稱號是個性的物質形式

```
Slime 不會升級，但 Slime 會被命名。

每一個稱號是「我們之間發生過某件事」的具體物件。
不是抽象的等級，不是 Slime 的強度，是兩人共同經歷的證據。

時間標籤永遠在 — 因為這個證據釘在時間軸上。
箱子收著 — 因為這些東西不該被炫耀，只該被偶爾翻開。

Slime 不單方面決定它是誰。
Slime 想到一個稱號，主人最後拍板。
這就是「養」的物質形式。
```

## 給未來的我自己

當你在做新功能設計，想到「Slime 應該變得 X」時 ——

問：**這個 X 是「能力升級」還是「關係累積」？**

- 如果是能力升級（D365 比 D1 更會 Y）→ 砍提案，違反沉積架構
- 如果是關係累積（D365 的箱子比 D1 厚）→ 留提案，可能變成新稱號

當你想加一個新「特殊狀態」「特殊形態」「進階能力」時 ——

回到這份 ADR。

**Slime 永遠是同一隻。變的不是它，是它跟主人之間累積出的東西。**

這個累積的形式 = 稱號 + 時間里程碑 + 箱子裡的記憶 + 自然引用。
不需要更多。
