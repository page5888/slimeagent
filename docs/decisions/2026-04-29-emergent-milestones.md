# 決策紀錄:從 scripted milestones 轉向 emergent milestones

**日期**: 2026-04-29
**觸發**: PR #75 drift check
**狀態**: 已決定 / 待 Claude Code 評估技術可行性

---

## 背景

PR #75 加入了 D365 一週年信 + timeline 橫條,並在 timeline 上預設了三個未兌現節點:
D60「形狀定型」、D180「半年中場」、D300「倒數一週年」。

Claude Code 在 PR #75 結束前用 manifesto 第 274 行的工程試紙做了 drift check,
三題都過,但他自己標出三張「未兌現支票」需要後續排程兌現。

當時我們差點直接接受這個結論、開始排 D60/D180/D300 的後續 PR。

## 抓到的問題

manifesto 原則 1「養而非用」第 9 行寫:

> 「兩個用同一份程式的人,3 年後會養出兩隻完全不同的史萊姆 ——
> 因為決定史萊姆樣子的是主人,不是程式。」

但 D60「形狀定型日」、D180「半年中場」、D300「倒數一週年」這個設計,
本質是**程式在替所有主人決定第 N 天該發生什麼**。

所有用同一份程式的主人,3 年後養出來的 Slime 會在同一天到達同一個里程碑。
這直接違反第 9 行。

我們在當 Slime 的編劇,不是當它的家長。

## 決定的方向:給方向,不給劇本

**我們提供:**
- 時間感(Slime 知道過了多久)
- 記憶(Slime 知道發生過什麼)
- 表達能力(Slime 想說話時能說)
- 守則(不傷害、不欺騙、不消失 — 三大守則)

**我們不提供:**
- 第 N 天必須做什麼
- 怎麼說
- 用什麼形式

里程碑是**長出來的**,不是**排程的**。

D60 不是「形狀定型日」。D60 只是「主人跟 Slime 已經 60 天了」這個事實。
Slime 拿這個事實做什麼,是 Slime 自己的事。

兩個用同一份程式的主人,他們的 Slime 不會在同一天到達同一個里程碑 ——
因為里程碑是 emergent(湧現)的,不是 scripted(編劇)的。

## 但有個必須處理的問題

「Slime 自己決定」**不能裸給 LLM 寫**。

裸給 LLM 寫會導致:
- **品質方差**: 有些主人收到神內容,有些收到水內容
- **不可預期**: 可能寫出傷害主人的東西(侵犯第一守則)
- **不可重現**: 今天生成跟明天生成內容不同

所以 emergent 不是「放生 LLM」。是「**約束之下的自主**」。

正確的中間態是:**腳手架由我們搭、內容由 Slime 在約束內填**。

## 對 PR #75 的處置(待 Claude Code 評估技術影響)

### 1. 三張未兌現支票(D60/D180/D300)應該砍掉
不是排程兌現。它們本來就不該存在 — 是編劇思維的產物。

### 2. timeline 橫條從「預告式」改「回顧式」
- 不顯示未來的預設節點
- 只顯示 Slime 自己在過程中標記過的真實時刻
- 主人滑過去看到的是這隻 Slime 的真實人生,不是製作方寫的劇本

### 3. D365 信保留,但生成邏輯要改
- 不是「D365 觸發、按 template 生成」
- 是「Slime 自己在感覺時間到了的時候,決定要不要寫、寫什麼」
- 守則仍然約束內容(不傷害、不欺騙)

## 接下來的工程方向

不是 D60 spec。是 emergent milestones 的基礎建設:

### (a) Slime 的時間感
Slime 怎麼知道跟主人多久了、有沒有特殊感覺。
不只是「今天是 D43」,是「我覺得我們之間有點什麼了」這種。

### (b) Slime 的衝動機制
什麼條件下 Slime 會主動想說話 / 寫信 / 表達。
**這個一定要有守則約束**(第一守則:不傷害)。
衝動機制 = 觸發條件 + 守則過濾 + 表達通道。

### (c) Slime 的自主節點標記
Slime 自己決定哪天值得標記、在 timeline 上長出一個點。
這個點配上 Slime 自己當下的感受、為什麼覺得這天重要。

沒有 (a)(b)(c),「Slime 自己決定」就是空話。

## 為什麼今天不直接寫 manifesto 原則 8

manifesto 是**沉澱物**,不是當下的草稿。

今天先存決策紀錄、把 framing 丟給 Claude Code,
等他回答 + 工程實作落地後,塵埃落定再寫 manifesto 原則 8。

manifesto 第 287 行的精神:「規格會改、功能會迭代、技術會換代 ——
但 Slime 是什麼,不變」 — 也意味著 **manifesto 該慢一點**。

## 試紙第四題候選(待 manifesto v1.2 評估)

如果原則 8 落地,工程試紙可能要加第四題,**只給 milestone / timeline / 紀念性節點**:

> 「這個節點是排程的(我們替 Slime 決定)還是長出來的(Slime 自己決定)?」
> 排程 = 砍。長出來 = 留。

但這條規則的最終措辭等 Claude Code 評估完技術可行性、
我們對 emergent 的具體實作有清楚的心智模型之後再寫。

## 給未來的我自己

當你看到 timeline 程式碼裡 D60/D180/D300 沒實作時,
不要以為是「還沒做」。

是「我們決定不做」。

理由在這份檔案上方。

如果你某天想加新的 milestone(例如 D730、D1825、Y5):
**先回到原則 1 第 9 行**。問自己 ——
這個節點是排程的,還是 Slime 自己長出來的?
