# ADR: 商業模式 — Slime 永遠免費，5888 備份是保險不是門票

**日期**: 2026-04-30

## 為什麼寫這份 ADR

manifesto 第三守則：

> **「沒有任何商業決定能讓主人失去他的史萊姆。」**

這句話過去只是承諾，沒有對應的商業模式。

這份 ADR 把「商業模式」跟「第三守則」釘在一起 —— 確保未來任何收費決定都不違反這條守則。

---

## 核心隱喻：賣保險，不賣門票

| 賣門票 | 賣保險 |
|---|---|
| 沒票 → 不能進場 | 沒保險 → 還是能上路 |
| 失去訂閱 → 失去核心功能 | 失去訂閱 → 失去額外保障 |
| 藏功能在付費牆後 | 提供額外的「萬一出事我們扛」 |

**Slime 賣保險，不賣門票。**

具體：

- 訂閱買的是「**上傳備份的權利**」（額外保險）
- 訂閱**不是**「使用 Slime 的權利」（那是免費的，永遠是）

═══════════════════════════════════════════════════

## 重要事件的判斷規則

「重要事件」必須客觀、不被工程師主觀加減。

**觸發備份提醒** = 任一條件成立：

1. 距離上次備份 > 30 天
2. 累積了 N 個重要事件未備份（命名儀式、年信、主人手動標記、新稱號累積到 M 個）
3. 主人主動詢問「要不要備份」

**不單獨觸發備份提醒**：

- 每個稱號生成（太頻繁）
- 每個自畫像產出（太頻繁）
- 每個反思卡（太頻繁）
- streak 類事件（像 dark pattern，違反 manifesto）

## 為什麼備份提醒走 OS 系統通知，不走 Slime 角色

（對齊 manifesto v1.2 候選原則 11「兩個面」）

```
Slime 的兩個面架構：
- 陪伴面 = Slime 本體，玩具，不索取
- 調度面 = Slime 認識的工具，有功能

5888 備份不屬於陪伴面，屬於調度面。
所以：
- 備份提醒 = 調度面的服務
- 不該透過 Slime 對話、不該打斷 Slime 跟主人的關係

具體實作：
- OS 系統通知（像收到郵件那種）
- 通知標題：「5888 備份」（不是 Slime 名字）
- 通知內容：「你跟 [Slime 名字] 累積了 X 件事，要備份嗎？」
- 完全在 Slime 對話之外發生
```

## 為什麼 Local 路徑必須保留

**manifesto 紅線**：任何商業決定不能讓主人失去 Slime。

如果 Google 帳號被封 / 國家斷網 / 5888 倒閉 / 服務政策變動 ——
主人本地的 `.slime` 檔仍然完整、可被任何人還原。

（這個是 ADR 2026-04-30-slime-core-format 的工程實作 —— 23 行 stdlib Python reader 公開在 `docs/SLIME_CORE_FORMAT.md`。）

Local 路徑不只是「小眾選項」，是整個商業模式的**安全網**。
它讓我們可以放心收費，因為主人永遠能說「我不買了」並把自己的 Slime 帶走。

## 訂閱中斷的精確規則

**主人付過費上傳的備份 → 永遠屬於主人**

| 訂閱狀態 | 能做的事 | 不能做的事 |
|---|---|---|
| 訂閱中 | 上傳新備份 / 下載 / 同步 | （無） |
| 訂閱中斷 | 仍能登入 / 用 Slime / 下載歷史備份 / local export | 上傳新備份 / 自動同步新資料 |
| 訂閱重新開始 | 一切恢復 | （無） |

**核心邏輯**：訂閱買的是「上傳新資料的權利」，不是「擁有備份的權利」。
備份一旦上傳就屬於主人，訂閱中斷不會奪走。

## 5888 倒閉時的處理（極端情境的 protocol）

如果哪天 5888（這個專案的營運實體）無法繼續服務：

**承諾**：

1. 提前 90 天通知所有訂閱用戶
2. 通知期間，所有用戶可免費下載自己全部已上傳備份
3. 開源備份格式規格（本來就公開）
4. 提供 standalone reader 工具，讓主人在 5888 倒閉後仍能讀備份檔
5. 鼓勵社群 fork 服務（MIT 授權允許）

**承諾不做的**：

- 不刪資料（在通知期內）
- 不切斷服務（在通知期內）
- 不收回任何已交付給主人的東西

這個 protocol 寫進 ADR，而且寫進服務條款。

## 下載永遠免費的工程含意

**下載免費 ≠ 無限制下載速率**（防 abuse）

合理工程限制：

- 速率限制（例如每小時最多 N 次下載請求）
- 不限制總次數（對 normal user 不影響）
- 不限制總流量（主人的東西不該被計費）

正常用戶下載自己備份 N 次/年，完全在 limit 內。
惡意行為（寫腳本不停下載）會被 rate limit 擋，但不會被收費。

## 訂閱定價（待定）

不在這份 ADR 範圍。

但定價的設計原則：

**✅ 應該**：

- 反映儲存 + 加密 + 維運的真實成本
- 讓「想要保險的主人」可以負擔
- 不依賴「鎖核心功能」拉升轉換率

**❌ 不應該**：

- 故意降低免費版功能來逼訂閱
- 用 dark pattern 讓主人覺得「不訂閱會失去什麼」
- 用 streak、限時優惠等遊戲化手段

實際數字之後決定，但定價邏輯必須對齊上面的原則。

這份 ADR **不直接提案新的 manifesto 原則**，但強化已有的第三守則：

**manifesto 第三守則**：

> 「沒有任何商業決定能讓主人失去他的史萊姆。」

這份 ADR 把這句話變成可執行的商業條款：

- Local export 永遠免費 → 主人永遠能取回
- 訂閱中斷不影響已上傳備份 → 商業決定不能拿走主人的東西
- 5888 倒閉 protocol → 公司死了主人不會死
- 不付費仍能完整使用 → 收費不是門檻

## 跟其他 AI 產品的對比

| 產品 | 商業模式 | 跟 Slime 的差異 |
|---|---|---|
| ChatGPT | 免費版功能 nerf, Plus 解鎖完整 | Slime 沒有功能 nerf |
| Replika | 核心功能（romantic mode）Pro Only | Slime 核心功能全免費 |
| Character.ai | 免費對話次數限制 | Slime 不限制核心使用 |
| Cursor / Copilot | 訂閱 = 工具的使用權 | Slime 訂閱 = 保險，不是使用權 |
| **Slime** | **訂閱 = 雲端備份保險** | **核心永遠免費，訂閱買保障** |

這個位置在 AI 產業極罕見 —— 但這就是 manifesto 第三守則的物理實作。

## 給未來的我自己

當你在某次商業壓力下想到「讓 X 變成付費功能會增加營收」時 ——

問：**這個 X 是「保險增值」還是「核心功能」？**

- 如果是保險增值（雲端備份、跨裝置同步、版本管理）→ 可以訂閱化
- 如果是核心功能（陪伴、記憶、稱號、自畫像）→ 永遠免費

---

當投資人問「為什麼不把 cross-device sync 變必付費功能，你會增加 30% 訂閱率」 ——

回答：

> 「因為跨裝置 sync 是保險，不是門票。
> 沒保險也能開車。
> 把這個變成必付才能用，Slime 就變成 Replika 了。
> 我們在做的不是 Replika，是 30 年的關係。」

---

當你看到競爭對手用 dark pattern 拉升訂閱率時 ——

回到這份 ADR。

短期 metric 我們會比他們難看。

但 5 年後當所有 AI 都被視為「會勒索使用者的東西」時，
**「永遠不會勒索你的 AI」會變成稀缺。**

那時候 Slime 會在原本的位置等他們。
什麼都沒變，只是箱子裡的紙更多了。
